# Manual verification — non-scriptable release checks

The automated `tools/cli/src/__tests__/e2e-walkthrough.test.ts` covers most of
the bootstrap walkthrough (init server → invite → init client → wire repo +
sidecars + git hooks). The steps below cannot be scripted — they require a
real IDE session, a real OS keychain, or `systemd` on Linux. Run them in
order on a freshly-bootstrapped install.

---

## 1. Subprocess `init client` against the real CLI

**Why manual.** The test runs `runInitClient()` in-process so it can stub the
keychain. The actual `node dist/index.mjs init client` invocation requires a
real OS keychain (libsecret on Linux, Keychain.app on macOS) which CI runners
do not provide.

```sh
# After running `brain init server` and copying the bootstrap admin PAT:
export BRAIN_AUTH_TOKEN=<bootstrap-admin-pat>
INVITE=$(brain admin invite --namespace verify-team --ttl 1h | head -1)

# In a fresh shell (no BRAIN_AUTH_TOKEN), as the joining user:
brain init client --invite "$INVITE" --server http://localhost:7430 --no-wire
```

**Expected.**
- `~/.second-brain/credentials/localhost:7430.json` exists, mode `0600`.
- `security find-generic-password -s second-brain -a "pat:localhost:7430:<tokenId>"`
  (macOS) or `secret-tool lookup service second-brain account "pat:..."`
  (Linux) returns the PAT.
- Stdout shows the credentials path but **does not** print the PAT (keychain
  success path).
- `brain doctor` reports `whoami → 200` with a `userId` matching the stored
  credentials (namespace `verify-team`). There is no `brain whoami` command —
  the identity endpoint is `GET /api/auth/whoami`; for raw fields run
  `curl -fsS -H "Authorization: Bearer <pat>" http://localhost:7430/api/auth/whoami | jq`.

## 2. Real Claude Code / Cursor / Codex / Copilot context injection

**Why manual.** Hook context injection happens inside the IDE process; we'd
need to drive each IDE programmatically to observe `additionalContext`.

For each adapter installed via `brain wire-assistant <name>`:

| Adapter | How to verify |
|---|---|
| Claude Code | Open the project in Claude Code. The first message in a new session should include a "Recent decisions, WIP and patterns" block sourced from the brain. Tail `~/.second-brain/hook.log` and look for `session-start ... adapter=claude` followed by `additionalContext` length > 0. |
| Cursor | Open Cursor in the project. Open the chat panel and ask "what am I working on?" — the answer should reference recent WIP. Cursor logs hook output to `~/Library/Logs/Cursor/main.log` (macOS); grep for `brain-hook`. |
| Codex | Run `codex` in the project. The opening turn should mention the team manifest. `~/.codex/log/codex.log` carries hook stdout/stderr. |
| Copilot | Open the repo in VS Code with Copilot Chat. Copilot reads `.github/hooks/second-brain.json` and `.github/copilot-instructions.md` (regenerated by brain-hook on session start). Verify the file timestamp is fresh. |

**Expected.** All four IDEs surface brain context within the first turn.

## 3. p95 hook latency in `~/.second-brain/hook.log`

**Why manual.** The hook log only accumulates after real hook firings; the
e2e test fires zero real hooks.

After 10–20 minutes of normal IDE usage in a wired repo:

```sh
# Each line of hook.log is a JSON record with a `latencyMs` field.
jq -s '
  map(select(.latencyMs != null) | .latencyMs) |
  sort | .[((length * 0.95) | floor)]
' ~/.second-brain/hook.log
```

**Expected.** p95 ≤ 250ms (hook budget per `tools/cli/src/hook-binary.ts`).
If higher: inspect the slowest entries (`jq -s 'sort_by(.latencyMs) | .[-5:]'`)
to see which adapter / event is slow.

## 4. Cap behavior at 9 vs 10 large injections in one session

**Why manual.** Requires forcing 10+ context-eligible tool calls in one
session, which is hard to script outside the IDE.

In Claude Code, inside a wired repo:

1. Open 10 different files in quick succession (`Read tool`).
2. Tail `~/.second-brain/hook.log` and grep for `cap=`:
   ```sh
   tail -f ~/.second-brain/hook.log | grep -E '"cap":(9|10|11)'
   ```

**Expected.**
- Injections 1–9: `cap=9` (or similar) — context injected.
- Injection 10+: `injected=false reason=cap-reached` — hook still returns
  within budget but contributes no `additionalContext`.

## 5. `systemd-analyze security` (Linux only)

**Why manual.** Requires Linux + systemd; macOS CI cannot run this.

```sh
# After `sudo systemctl daemon-reload && sudo systemctl enable --now second-brain-server`:
systemd-analyze security second-brain-server
```

**Expected.** Overall exposure level `MEDIUM` or better (numeric score < 3.0).
The Tier-1 hardening directives in `tools/cli/src/init-server.ts:renderSystemdUnit`
were chosen to hit this target on Ubuntu 22.04+ / Debian 12+; if your distro
ships an older systemd that doesn't recognize one of the directives, you'll
see `unsupported` warnings — those are informational, not failures.

## 6. `brain auth rotate` end-to-end with a live server

**Why manual.** The unit tests cover the rotate flow against a fake fetch.
A real-server check confirms the round-trip + keychain delete + credentials
patch all work against the production code path.

```sh
# 1. Note the current token id from the credentials file.
cat ~/.second-brain/credentials/<host>.json | jq '{hookTokenId, defaultTokenId, cliTokenId}'

# 2. Rotate.
brain auth rotate

# 3. The CLI prints the slot that fired (e.g. "rotating slot 'hook'") and the
#    new token id. Confirm the credential still authenticates:
brain doctor     # whoami → 200, userId unchanged; rotated slot's tokenId updated

# 4. The OLD keychain entry is gone:
security find-generic-password -s second-brain -a "pat:<host>:<oldTokenId>"
# → expected: "could not be found"

# 5. The credentials file shows the new tokenId in the originating slot
#    only — the other two slots are unchanged:
cat ~/.second-brain/credentials/<host>.json | jq '{hookTokenId, defaultTokenId, cliTokenId}'
```

**Expected.** Steps 4 and 5 succeed; only the rotated slot's pointer changed.
For env-source rotation (`BRAIN_AUTH_TOKEN=...; brain auth rotate`), the CLI
exits with code **2** and prints an `export BRAIN_AUTH_TOKEN=<new-pat>`
instruction.
